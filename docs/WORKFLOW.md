# Cel Band Pack — PNG production pipeline

> **Distribution note:** Shipped as **Cel Band Pack** with franchise terms removed from manifest and docs. See `docs/TRADEMARK_AUDIT.md`.

This document describes exactly how every PNG in `Lara_Croft_GO_Diorama.mcpack` was produced. There are no hand-authored image files; every pixel is generated by `lcgo_mc_tool.py` using PIL/Pillow 11.3+.

## Toolchain

| Component | Version | Purpose |
|---|---|---|
| Python | 3.12+ | Runtime |
| Pillow (PIL) | 11.3.0 | PNG generation and manipulation |
| Standard library (`zipfile`, `json`, `uuid`, `random`, `pathlib`) | builtin | Pack assembly, manifest generation, deterministic seeding |
| ReportLab | 4.4.9 | Spec PDF generation (separate from PNG pipeline) |

No external image editor (Photoshop, Aseprite, Krita, GIMP) was used at any point. The Python tool is the sole source of truth.

## Source Format

There is no source image format. The "source" is the `PALETTE` dict at the top of `lcgo_mc_tool.py`:

```python
PALETTE = {
    'lit':         (244, 228, 193),  # warm cream (sunlit top face)
    'light_mid':   (212, 184, 150),  # warm sand (sunlit side face)
    'dark_mid':    (138, 111,  79),  # warm brown (occluded side)
    'shadow':      ( 74,  53,  40),  # deep warm brown (caves / underhangs)
    'ink':         ( 26,  15,   8),  # near-black warm (outlines)
    'bronze':      (184, 136,  74),  # LC GO tomb gold accent
    # ... 18 more colors
}
```

Edit this dict, rerun `--mode all`, and every PNG regenerates with the new palette. This is the entire iteration loop.

```

        │

        │       │
        │       ├─→ make_block_texture()       [for calcite, stone, deepslate, bedrock, sand, dirt, cobblestone]
        │       ├─→ make_log_top_texture()     [for oak_log_top]
        │       ├─→ make_log_side_texture()    [for oak_log_side]
        │       ├─→ make_leaves_texture()      [for leaves, leaves_opaque]
        │       ├─→ make_water_texture()       [for water]
        │       └─→ make_relic_block_texture() [for gold_block, emerald_block, copper_block]
        │               │
        │               └─→ 16x16 RGB PNG saved to textures/blocks/<vanilla_path>.png
        │
        ├─→ ITEM_TEXTURES dict (per-item specs)
        │       │
        │       ├─→ make_incan_bottle()    [silhouette-based, RGBA]
        │       ├─→ make_jade_mask()       [silhouette-based, RGBA]

        │       └─→ make_spear_item()      [silhouette + edge-detect outline, RGBA]
        │               │

        │
        ├─→ UI generators

                ↓
        PIL Image.save() to disk at vanilla path
                ↓

                ↓
```

## Cel Band Application

The 4-step cel band look is applied at texture-generation time, not at runtime. Each of the 4 bands uses a different vanilla block type:

| Band | RGB color | Vanilla block | Texture path |
|---|---|---|---|
| 1 (lit) | (244, 228, 193) warm cream | calcite | textures/blocks/calcite.png |
| 2 (light-mid) | (212, 184, 150) warm sand | stone | textures/blocks/stone.png |
| 3 (dark-mid) | (138, 111, 79) warm brown | deepslate | textures/blocks/deepslate/deepslate.png |
| 4 (shadow) | (74, 53, 40) deep brown | bedrock | textures/blocks/bedrock.png |

The band assignment is a build-time decision: the level converter (see `LCGO_TILE_TO_MC` dict) places the appropriate block type per tile, and the texture pack renders that block with the corresponding band color. There is no in-shader banding.

## Holstein Spot Algorithm

The irregular spot pattern on each block texture is generated by `holstein_spot_mask()`:

1. Seed a `random.Random(seed)` instance with the block's unique seed (calcite=11, stone=22, etc.).
2. For each of N spots (2-4 per block, configurable in `BLOCK_TEXTURES`):
   - Pick a random center (cx, cy) within the 16×16 grid, biased 2 pixels from edges.
   - Sample 8 perturbed radii around the center, each `uniform(min_r, max_r)`.
   - For every pixel in the 16×16 grid, check if its distance from (cx, cy) is less than the perturbed radius at that angle.
   - If yes, add the pixel to the spot set.
3. Composite the spot set onto the base color using `Image.putpixel()`.

This produces organic blobby shapes — not perfect circles — matching the graphic-illustration feel of LC GO's concept art.

## Ink Outline Algorithm

The 1px ink border is applied last, after the base color, noise, and spots:

```python
ink = PALETTE['ink']  # (26, 15, 8)
for i in range(SIZE):  # SIZE = 16
    img.putpixel((i, 0), ink)           # top edge
    img.putpixel((i, SIZE - 1), ink)    # bottom edge
    img.putpixel((0, i), ink)           # left edge
    img.putpixel((SIZE - 1, i), ink)    # right edge
```

Water is the only block without an ink border (it flows visually). Items use a different outline approach: edge detection on the alpha channel (`make_spear_item()` is the reference implementation).

## Export Settings

| Property | Value |
|---|---|
| Format | PNG (8-bit) |
| Block textures | RGB, 16×16, no alpha |
| Item textures | RGBA, 16×16, with alpha for transparent regions |
| UI textures | RGBA, 16×16 or 64×64 |
| Pack icon | RGB, 256×256 |
| Compression | Pillow default (zlib level 6) |
| Color profile | None (sRGB assumed, no embedded profile) |

No dithering, no anti-aliasing on the spot edges (intentional — keeps the graphic-illustration look). The only anti-aliasing in the pipeline is implicit in Pillow's `Image.putpixel()` boundary handling.

## Bedrock Path Verification

Every block texture path was verified against the canonical Bedrock vanilla resource pack at `https://github.com/ZtechNetwork/MCBVanillaResourcePack` (master branch, fetched 2026-06-22).

Key findings from the verification (these were bugs in v1.0.0, fixed in v1.1.0):

| Block | v1.0.0 path (WRONG) | v1.1.0 path (CORRECT) | Reason |
|---|---|---|---|
| deepslate | textures/blocks/deepslate.png | textures/blocks/deepslate/deepslate.png | Bedrock uses a subdirectory |
| oak_log top | textures/blocks/log_top.png | textures/blocks/log_oak_top.png | Bedrock naming is "log_oak_top" not "oak_log_top" or "log_top" |
| oak_log side | textures/blocks/log_side.png | textures/blocks/log_oak.png | No "_side" suffix in vanilla; just "log_oak" |
| water | textures/blocks/water_still_greyscale.png | textures/blocks/water_still_grey.png | British spelling "grey", not "greyscale" |
| leaves | (only fancy variant) | textures/blocks/leaves_oak.png + leaves_oak_opaque.png | Must override both fancy and fast/opaque variants |

The path resolution chain in Bedrock is:
1. `blocks.json` maps a block name (e.g. `oak_log`) to texture keys (e.g. `oak_log_top`, `oak_log_side`).
2. `textures/terrain_texture.json` maps those keys to file paths (e.g. `oak_log_top` → `textures/blocks/log_oak_top`).
3. The resource pack must place PNGs at the file paths from step 2, not at paths derived from the block name.

This is why naive "block-name → path" mappings fail for logs, water, and deepslate.

## Regeneration Workflow

To regenerate all PNGs and reassemble the pack:

```bash
cd /home/z/my-project

# 1. Edit the PALETTE dict at the top of scripts/lcgo_mc_tool.py
#    (or edit BLOCK_TEXTURES seeds, spot counts, etc.)

# 2. Regenerate everything
python3 scripts/lcgo_mc_tool.py --mode all --output ./download/lcgo_mc_output

# 3. Inspect the generated PNGs (optional)
python3 -c "from PIL import Image; img = Image.open('download/lcgo_mc_output/textures/blocks/stone.png'); img.show()"

# 4. The .mcpack is automatically assembled at:
#    /home/z/my-project/download/Lara_Croft_GO_Diorama.mcpack
```

To regenerate only textures (skip pack assembly):

```bash
python3 scripts/lcgo_mc_tool.py --mode textures --output ./download/lcgo_mc_output
```

To regenerate only the pack (after manual texture edits):

```bash
python3 scripts/lcgo_mc_tool.py --mode pack --output ./download/lcgo_mc_output
```

## Determinism

Every PNG is deterministic given the same `lcgo_mc_tool.py` source. The `random.Random(seed)` calls use per-block-type seeds (calcite=11, stone=22, deepslate=33, bedrock=44, sand=55, dirt=66, cobblestone=67, oak_log_top=77, oak_log_side=88, leaves=99, water=110, gold_block=121, emerald_block=132, copper_block=143). Changing any seed produces a different spot pattern; changing the palette produces different colors with the same spot pattern.
